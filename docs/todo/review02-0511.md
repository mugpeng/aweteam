# aweswitch Review ‚ÄĒ Final Summary

**Scope**: Architecture, CLI/UX, bugs & test gaps across `aweswitch.py` (~237 lines) and `test_aweswitch.py`.

**Overall**: Well-scoped, clean single-file CLI with good separation of concerns, zero external dependencies, and solid test coverage for happy paths. No blocking issues. Findings below are ranked by impact.

---

## High-Value Findings

### 1. Codex provider leaks secrets into child env (Medium)
`aweswitch.py:107-108` ‚ÄĒ Codex profiles inject all env vars (including `OPENAI_API_KEY`) directly into `os.execvpe`. Claude profiles use `--settings` for isolation, but the Codex path has none. Any subprocess codex spawns inherits all secrets.
**Fix**: Pass secrets via a mechanism analogous to `--settings`, or document this as intentional scope.

### 2. Redaction hides env-var references (Medium)
`aweswitch.py:120-135` ‚ÄĒ `aweswitch show` redacts `${OPENAI_API_KEY}` references to `<redacted>`, making it impossible to audit whether a profile references a var or hardcodes a secret.
**Fix**: Skip redaction when the value matches the `${‚Ä¶}` pattern.

### 3. Malformed profiles crash instead of reporting errors (Medium)
`aweswitch.py:152-163` ‚ÄĒ `command_list()` and `profile_model_label()` assume profile values are dicts. A config like `{"profiles": {"bad": []}}` raises `AttributeError` instead of a `die()` message.
**Fix**: Validate profile shape during loading.

### 4. Non-string env values crash at exec time (Medium)
`aweswitch.py:106-108` ‚ÄĒ `prepare_run()` copies env values without type validation. A typo like `OPENAI_API_KEY: 123` passes silently until `os.execvpe` raises `TypeError`.
**Fix**: Validate env values are strings during profile loading.

### 5. Silent settings.json parse failure (Medium)
`aweswitch.py:59-60` ‚ÄĒ `load_claude_settings_env` returns `{}` on invalid JSON with no warning. Profile expansion then fails with confusing "missing variable" errors.
**Fix**: Warn on stderr when JSON parse fails.

## Test Coverage Gaps

Current tests (11 cases) cover happy-path profiles and one missing-variable error. Missing coverage:
- Malformed profile shapes (non-dict values)
- Non-string env values
- Invalid JSON in `~/.claude/settings.json`
- Codex provider execution path
- Redaction logic with `${VAR}` references

## Recommended Priority

1. Validate profile shapes and env types during loading (fixes findings 3 + 4, enables better error messages)
2. Fix redaction to expose `${VAR}` references (finding 2)
3. Add stderr warning for settings.json parse failure (finding 5)
4. Consolidate provider dispatch into one place (findings 6 + 7) if a third provider is planned
5. Flesh out test coverage for the gaps above
