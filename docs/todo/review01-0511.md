# aweswitch Project Review

**Project:** CLI launcher (~240 lines, single-file Python) that switches between Claude Code and Codex runtime profiles via JSON config.

**Reviewers:** codex-gpt5.4-mini (architecture), cc-glm (security), cc-gemini (code quality)

---

## Critical & High Findings

### 1. CRITICAL — Secrets visible via `ps` for Claude profiles
**File:** `aweswitch.py:104`

Expanded secrets (API tokens) are passed as a command-line argument in `--settings`. Any local user can read them via `ps aux` or `/proc/<pid>/cmdline`. The Codex path correctly uses `env[]` instead.

**Fix:** Write settings to a temp file with mode 600, or use an fd-based mechanism.

### 2. HIGH — Config file created world-readable
**File:** `aweswitch.py:37`

`init_config` uses `shutil.copy2` which preserves 644 permissions. Users who embed real tokens will have them world-readable.

**Fix:** Add `path.chmod(0o600)` after copy.

### 3. HIGH — User args can inject `--settings` override
**File:** `aweswitch.py:105`

User arguments are appended after the `--settings` flag. If Claude Code processes the last `--settings` as authoritative, an attacker controlling args can inject a malicious token:

```
aweswitch cc-glm --settings '{"env":{"ANTHROPIC_AUTH_TOKEN":"attacker"}}'
```

**Fix:** Validate that `user_args` does not contain `--settings`, or pass settings via a non-overridable mechanism.

---

## Medium Findings

### 4. Config validation is too permissive
`load_config` only checks that `profiles` is a dict — it doesn't validate profile shape, `provider` field, or `env` type. Bad profiles produce confusing downstream errors instead of clear config errors.

### 5. Silent swallow of malformed `~/.claude/settings.json`
`load_claude_settings_env` returns `{}` on JSON parse failure. The user gets a misleading "missing environment variable" error instead of being told their settings file is corrupt.

### 6. Redaction regex too narrow
`SECRET_RE` matches `TOKEN|KEY|SECRET|PASSWORD|AUTH` but misses `CREDENTIAL`, `PRIVATE`, `DATABASE_URL`, `CONNECTION_STRING`, etc.

### 7. Duplicate `if provider == "claude"` blocks in `prepare_run`
Lines 96 and 100 check the same condition for different purposes (expansion_env vs argv). Should be combined for clarity. As providers grow, consider a simple handler registry.

---

## Low / Code Quality Findings

- **No type hints** on any function signature — limits static analysis value.
- **No docstrings** — acceptable at this size but would help on public functions.
- **JSON-loading logic duplicated** in `load_config` and `load_claude_settings_env` — extract a helper.
- **Path expansion duplicated** across `init_config`, `load_config`, `load_claude_settings_env`.
- **Missing env var context in errors** — `die("missing environment variable: X")` doesn't say which profile or key triggered it.
- **No file-read error handling** — `PermissionError` on config files bubbles up as a raw traceback.

---

## Test Coverage Gaps

- No tests for `main`, `command_config`, `command_list`, `command_show`.
- No tests for error conditions (invalid JSON, missing profiles, malformed profiles).
- `exec_agent` only tests `FileNotFoundError`, not `OSError`.
- No integration test verifying the full `load → prepare → exec` pipeline.

---

## Architecture Notes

The single-file approach is appropriate for ~240 lines. The main architectural debt is:

1. **Provider branching** is scattered across `prepare_run` and `profile_model_label`. A small provider-handler table (dict of name → callable) would make adding a third provider a single-location change.
2. **Config schema** has no validation layer — adding a lightweight schema check at load time would prevent most downstream issues.
